# 前后端对接规范

前后端通过**统一 JSON 响应体**通信。本规范确保前端不会错误对接。

## 1. 统一响应体

后端所有接口返回相同结构，由 `shared/utils/response` 处理：

```json
{
  "code": 0,           // 业务码：0=成功，非0=失败（见 errorx 码表）
  "msg": "success",    // 面向用户的消息
  "data": { ... }      // 业务数据（仅成功时存在；省略而非 null）
}
```

**前端必须按此结构统一解析**，不在每个调用处各写各的判断。

## 2. 状态判断

| 条件 | 含义 | 前端行为 |
|------|------|---------|
| `code === 0` | 成功 | 取 `data` 渲染 |
| `code !== 0` | 业务错误 | 取 `msg` 展示；按 code 做差异化处理（如 401 跳登录） |
| HTTP 非 200 | 网络/网关错误 | 通用错误提示 |

> HTTP 状态码统一为 200。**不要按 HTTP 状态判断业务结果。**

## 3. 错误码（errorx 集中码表）

| Code | 含义 | 前端常见处理 |
|------|------|-------------|
| 0 | 成功 | — |
| 40000 | 参数非法 | 表单校验提示 |
| 40100 | 未认证/token 失效 | 跳登录页 |
| 40300 | 无权限 | 提示无权限 |
| 40400 | 资源不存在 | 提示不存在 |
| 40900 | 资源冲突 | 提示冲突（如重复提交） |
| 42900 | 请求过于频繁 | 提示稍后重试 |
| 50000 | 服务内部错误 | 通用错误提示 |

> 各项目可在 `errorx` 中扩展 5xx 域内码，保持前后端码表同步。

## 4. 响应拦截器（前端必做）

每个前端项目必须封装统一的响应拦截器，**不依赖开发者自觉**：

- 解析 `{code, msg, data}`。
- `code === 0`：将 `data` 传给业务代码。
- `code !== 0`：统一错误处理（toast/弹窗 + 按 code 跳转）。
- 401：自动刷新 token 或跳登录。
- 网络错误：兜底提示。

## 5. 请求约定

| 项 | 规范 |
|----|------|
| Content-Type | `application/json`（body 为 JSON） |
| 鉴权 | `Authorization: Bearer <token>` |
| 路径前缀 | `/api/v1/front/...`（前台）、`/api/v1/admin/...`（后台） |
| 分页 | 由 `.api` 定义，常见 `page`/`page_size` 或 `offset`/`limit` |

## 6. 流式（AI/实时场景）

若后端返回 SSE/WebSocket 流：

- 流式数据**不走统一响应体**，直接是增量文本。
- 前端用 `EventSource` 或 WebSocket 接收，渲染到对应组件。
- 结束时后端可通过 SSE 附带一个最终 `{code, msg, data}` 事件，前端用统一错误处理。

## 7. 文件上传

- 后端先返回 `storage.PresignResult`（预签名 URL），前端直接上传到对象存储。
- 上传完成后前端把 `object_key` 传给后端接口。
- **前端不直接把文件 POST 到后端**（避免后端做代理中转）。

## 8. 前后端协作流程

- **tech-lead** 在 `design.md` 中定义接口契约（URL、请求/响应结构、错误码）。
- **ui-designer** 基于设计产出组件/页面，标注状态（加载/成功/错误）。
- **frontend-dev** 按契约实现前端，对接统一响应体。
- **对接检查**：由 `code-reviewer` 或 `tech-lead` 确认前端按本规范实现。
