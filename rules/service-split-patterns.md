# 微服务拆分模式

本文件指导 tech-lead 评估单体服务是否应该拆分为微服务，以及如何拆。
它不是教条——不是"微服务更好"，而是"什么条件下拆能带来价值"。

## 核心原则

**不为了"微服务"而拆。** 微服务带来独立部署、独立扩容、独立团队的好处，
也带来网络开销、数据一致性、调试复杂的成本。只有收益大于成本时才拆。

---

## 症状 1：部署频率不一致

**表现：**
- 模块 A（如 order 模块）每周发布 3 次，总是因为修 bug 或加功能。
- 模块 B（如 user 模块）每月发 1 次，稳定很少改。
- 每次 A 发版，B 也得一起部署——B 本不需要。

**判断：**
A 和 B 的发布频率差距超过 3 倍，且 A 的发布节奏不受 B 约束。

**策略：**
把 A 独立成服务。B 留在单体或独立部署。
拆完后 A 只影响自己的部署，不影响 B。

**工作室实现：**
A 从单体中的 `{module}/` 目录，拷贝为独立服务根目录，
加上独立入口 `cmd/{module}-api/main.go`。`deploy/nginx/` 加一条转发规则。

---

## 症状 2：并发/资源需求差异大

**表现：**
- order 模块的 QPS 是 user 模块的 50 倍。
- 为了撑住 order，user 被迫扩容到不需要的资源。
- 或：数据库连接池被 order 占满，user 的请求排队。

**判断：**
两个模块的资源消耗差异 > 10 倍，且难以通过缓存/优化缩小差距。

**策略：**
把高资源模块独立部署，单独配并发/数据库连接池/实例数。
低资源模块保留原样。

**工作室实现：**
独立模块的 `deploy/docker/` 单独构建镜像，`deploy/compose/` 单独编排。
共享数据通过 API 调用（rpc）或异步消息（mq）通信。

---

## 症状 3：模块间出现循环依赖

**表现：**
- user 调 order，order 通过 callback 调 user。
- 或编译层面出现 import cycle（A → B → C → A）。

**判断：**
存在双向依赖；或周期依赖已经导致编译困难、部署受阻。

**策略：**
先解决依赖再考虑拆分。解决手段：
1. **提取共享逻辑**：把 user 和 order 都依赖的计算/校验逻辑提到 `pkg/` 下的独立包。
2. **引入异步事件**：把 order 的 callback 改为投递 MQ 消息，user 异步消费。
3. **引入 API 层**：将周期依赖中的一端改为调用另一端 API。

依赖消除后，按域拆出独立服务。

**工作室实现：**
- 共享逻辑放 `backend/shared/pkg/`。
- 异步事件用 `taskqueue` 抽象（add-worker-task 技能）。
- API 调用用 `rpc` 模块（gozero-add-api + gen-rpc.sh）。

---

## 症状 4：团队碰撞

**表现：**
- 两个团队在同一个 `go.mod` 中改动。
- 合代码时频繁冲突。
- 一个团队的 PR 因另一个团队的变更被 block。

**判断：**
两个团队的改动边界重叠超过 20% 的提交，且难以通过代码规范解决。

**策略：**
按团队边界拆分服务。每个团队拥有自己服务的全部代码。
共享 `shared/pkg/` 由核心团队维护，对外发布稳定版本。

**工作室实现：**
每个团队的模块独立为服务。共性走前端约定的 API 契约，
而非共享 Go 代码（Team A 不再直接 import Team B 的 model）。

---

## 症状 5：技术栈差异化需求

**表现：**
- 某个模块需要使用不同的数据库（如 order 需要 ES 搜索）。
- 某个模块需要不同的消息队列（如 payment 需要 Kafka 保证投递）。
- 某个模块的算法/模型需要 Python 而非 Go。

**判断：**
单一技术栈的约束已在拖累某个模块的发展。

**策略：**
该模块独立为服务，允许它使用最合适的技术栈。
共享数据通过 API/事件通信。

**工作室实现：**
该模块独立部署（非 Go？新起项目放在仓库顶层平铺），
不强制共享 `shared/pkg/`，只靠 API 契约通信。

---

## 拆分实践指南

### 第一步：识别边界（1-2 天）

```
1. 找到周期依赖 → 用症状 3 的判断方法
2. 标出数据域 → 每个模块的唯一表/共享表
3. 识别通信模式 → 同步 API / 异步 MQ / 事件
4. 画出目标状态 → 单体 → 新服务架构图
```

### 第二步：拆分项目（1-2 天）

```
1. 新建服务目录（仓库顶层平铺或 backend/{new-service}/）
2. 复制模块的 api/model/mq/cron 代码
3. 替换 import 路径为新 module
4. 独立 go.mod
5. go build ./... 通过
```

### 第三步：切割数据（2-5 天，最耗时）

单体数据库拆分到独立数据库的关键：
```
1. 识别该模块独有的表（独占表）
2. 识别跨模块共享的表（共享表/参照表）
3. 独占表 → 新数据库，通过 API 提供访问
4. 共享表 → 用事件同步或 API 获取
```

拆分完成前，可以让新服务继续读老数据库（共享实例），
迁移期后再隔离数据库。

### 第四步：部署上线（1 天）

```
1. deploy/docker/ 新服务 Dockerfile
2. deploy/nginx/ 加转发规则
3. deploy/compose/ 加新服务
4. CI 流水线加新服务构建
5. 灰度 → 全流量切换
```

---

## 什么时候不要拆

- **团队 1-5 人，模块 < 6 个** → 单体完全够用。
- **用户数 < 1K DAU** → 优化单体比拆微服务更快。
- **模块间频繁互调用、数据强一致要求** → 拆后的网络开销和最终一致性让你痛。
- **组织没有独立运维能力** → 微服务需要更多 infra 支持。
